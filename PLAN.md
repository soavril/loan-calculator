## 1. 한국 대출 상환방식 정의

### 1.1 핵심 용어 정리 (공공기관/은행 기준)

| 용어 | 정의 | 월 납부 특성 |
|------|------|-------------|
| **원리금균등상환** | 매월 원금+이자 합계가 동일 | 초기 이자 비중 높음, 후기 원금 비중 높음 |
| **원금균등상환** | 매월 원금이 동일, 이자는 잔액에 따라 감소 | 초기 납부액 최대, 점점 감소 |
| **만기일시상환** | 대출 기간 동안 이자만 납부, 만기에 원금 전액 상환 | 월 납부액 최소 (이자만), 만기 부담 최대 |
| **거치기간** | 원금 상환 없이 이자만 납부하는 기간 | 거치 종료 후 선택한 상환방식으로 전환 |

### 1.2 거치기간 상세

**정의** (한국주택금융공사, 시중은행 공통):
- 거치기간 동안: 원금 상환 없음, 이자만 납부
- 거치기간 종료 후: 남은 기간 동안 원리금균등 또는 원금균등으로 전환
- 예시: 총 30년 대출, 거치 2년 → 2년간 이자만 → 28년간 원리금균등

**계산 영향**:
- 거치기간 이자 = 대출원금 × 월이자율
- 상환기간 = 총 대출기간 - 거치기간
- 총 이자 증가 (거치기간 동안 원금 미상환으로 인해)

### 1.3 DSR (총부채원리금상환비율)

**참고 정보만 제공** (Phase 2):
- DSR = (모든 대출의 연간 원리금 상환액) / 연소득 × 100
- 일반적 규제: 40~50% 이내 (대출 종류/지역에 따라 상이)
- ⚠️ "참고용 정보이며, 실제 대출 심사와 다를 수 있음" 명시 필수

## 2. 3-Agent 토론 및 의사결정

### Agent A: Senior PM / Growth Marketer

#### 가장 중요한 것
1. **차별화 없으면 트래픽 없음**: 네이버/카카오뱅크 계산기와 동일하면 의미 없음
2. **A/B 비교가 핵심 USP**: "두 대출 비교"는 경쟁사 대부분 미지원
3. **Break-even 분석**: "금리가 얼마나 올라야 손해?"는 고객의 실제 고민
4. **거치기간 지원**: 대부분 계산기가 숨기거나 복잡하게 처리

#### 리스크
- SEO 키워드 "대출 계산기"는 은행/핀테크가 점령 → 롱테일 필수
- 네이버 검색 점유율 감소 중이지만 여전히 40%+ → 네이버 최적화 병행 필요
- 얇은 콘텐츠는 구글 HCU에 취약

#### 권장사항
- **MVP 차별화**: A/B 비교 + 거치기간 지원 + 공식 투명 공개
- **콘텐츠**: 각 상환방식별 별도 페이지 (500자 이상 고유 콘텐츠)
- **SEO**: 구글 + 네이버 동시 최적화, FAQ Schema 필수

---

### Agent B: UX/UI Designer

#### 가장 중요한 것
1. **모바일 85%**: 한 손 조작, 48px 터치 타겟 필수
2. **3초 이해**: 차이(Δ)가 즉시 눈에 들어와야 함
3. **입력 최소화**: 상환방식 선택 시 불필요한 필드 숨김
4. **숫자 가독성**: 한국식 단위 (억/만원) 표기 옵션

#### 리스크
- A/B 비교 UI가 복잡해지면 이탈률 증가
- 거치기간 입력이 추가되면 인지 부하 증가
- 상환 스케줄 테이블이 모바일에서 스크롤 지옥

#### 권장사항
- **레이아웃**: Loan A/B 탭 전환 (모바일) vs 좌우 병렬 (데스크탑)
- **결과**: Hero 요약 (차이 금액) → 상세 펼치기
- **상환표**: 기본 숨김, "상환 스케줄 보기" 버튼으로 펼침
- **Progressive Disclosure**: 거치기간은 "고급 옵션" 토글 내 배치

---

### Agent C: Senior Frontend Engineer

#### 가장 중요한 것
1. **계산 정확성**: 원 단위 반올림 규칙 명확화 (은행마다 상이)
2. **성능**: Core Web Vitals (LCP < 2.5s, FID < 100ms, CLS < 0.1)
3. **엣지 케이스**: 0% 금리, 0개월 거치, 거치기간 > 대출기간 등
4. **유지보수**: 상환방식별 계산 로직 모듈화

#### 리스크
- 원리금균등 공식의 부동소수점 오차 (30년 대출 시 수백 원 차이 가능)
- 상환 스케줄 테이블 360행 → DOM 성능 이슈
- 한국어 숫자 포맷팅 (억/만) 구현 복잡도

#### 권장사항
- **반올림**: 모든 월 납부액은 원 단위 반올림, 마지막 회차에서 조정
- **상환표**: Virtual Scroll 또는 Lazy Load (50행씩)
- **공식 분리**: `calculateEqualPrincipalInterest()`, `calculateEqualPrincipal()` 등 함수 분리
- **테스트**: 은행 계산기 결과와 비교 검증 (±100원 이내 허용)

### 갈등 해결 (Conflict Resolution)

| 쟁점 | Agent A (PM) | Agent B (UX) | Agent C (Dev) | **최종 결정** |
|------|-------------|--------------|---------------|--------------|
| A/B 비교 UI | 필수 | 탭 vs 병렬 고민 | 구현 가능 | ✅ 모바일: 탭, 데스크탑: 병렬 |
| 거치기간 | MVP 포함 | 고급 옵션 숨김 | 계산 복잡도 중간 | ✅ "고급 옵션" 토글 내 배치 |
| Break-even | MVP 포함 | UX 복잡 우려 | 역산 로직 필요 | ⚠️ **Phase 1.5로 연기** (MVP+2주) |
| 상환 스케줄 | SEO 가치 있음 | 모바일 UX 문제 | Virtual Scroll | ✅ 기본 숨김, 펼치기 버튼 |
| 숫자 포맷 | 억/만원 필요 | 가독성 ↑ | 구현 복잡 | ✅ 1억 이상만 "X억 Y만원" 표기 |

## 6. 계산 로직 명세

### 6.1 원리금균등상환 (Equal Total Payment)

**월 납부액 공식**:
```
M = P × [r(1+r)^n] / [(1+r)^n - 1]

P = 대출원금
r = 월 이자율 (연이자율 / 12 / 100)
n = 총 납부 회차 (개월)
M = 월 납부액
```

**총 이자**:
```
총 이자 = (M × n) - P
```

**엣지 케이스**:
- r = 0 (무이자): M = P / n
- n = 0: 오류 처리

### 6.2 원금균등상환 (Equal Principal)

**월 납부액 공식** (k회차):
```
월 원금 = P / n
k회차 잔액 = P - (월 원금 × (k-1))
k회차 이자 = k회차 잔액 × r
k회차 납부액 = 월 원금 + k회차 이자
```

**총 이자**:
```
총 이자 = Σ (k회차 이자), k=1~n
       = P × r × (n+1) / 2
```

### 6.3 만기일시상환 (Bullet Repayment)

**월 납부액** (만기 전):
```
M = P × r (이자만)
```

**만기 시 납부액**:
```
만기 납부 = P + (P × r) = P(1 + r)
```

**총 이자**:
```
총 이자 = P × r × n
```

### 6.4 거치기간 포함 계산

**거치기간 동안** (g개월):
```
월 납부액 = P × r (이자만)
거치기간 이자 합계 = P × r × g
```

**거치 후 상환기간** (n - g 개월):
```
원리금균등 또는 원금균등 공식 적용
단, n' = n - g (남은 기간)
P' = P (원금 동일, 거치 중 원금 상환 없음)
```

### 6.5 반올림 규칙

| 항목 | 규칙 | 비고 |
|------|------|------|
| 월 납부액 | 원 단위 반올림 | Math.round() |
| 이자 계산 | 소수점 유지 후 최종 합산 시 반올림 | 오차 최소화 |
| 마지막 회차 | 잔액 정산 (총액 - 기납부액) | 오차 조정 |

### 6.6 검증 기준

은행 계산기 대비 **±100원 이내** 허용
- 참고: 은행별 일수 계산, 반올림 시점 상이로 완전 일치 불가능
- FAQ에 "은행 계산과 다를 수 있는 이유" 명시
